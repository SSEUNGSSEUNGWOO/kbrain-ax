# 기존 시스템 운영 분석 및 인수 요구사항

> 대상: aicapa.kr (온라인 시험) + evalu-lens (서면평가)
> 작성 목적: 문제점 정리, KBrain-AX(가제) 전환 근거, 기존 운영사에 요구할 항목 정의

---

## 1. 기존 운영 문제점

### 1-1. 구조적 문제: 시스템 분리 운영

| 기능 | 시스템 | 문제 |
|------|--------|------|
| 서면평가 | evalu-lens | 평가 결과를 별도로 관리, 수동 취합 필요 |
| 온라인 시험 | aicapa.kr | 시험 결과를 별도로 관리 |
| 통합 현황 | 없음 | 두 시스템 데이터를 엑셀 등으로 수동 병합하여 최종 합격자 선정 |

두 시스템이 독립 운영되면서 다음과 같은 **운영 비효율**이 구조적으로 발생합니다:

- **휴먼에러**: 두 시스템 결과를 수동으로 병합하는 과정에서 합격자 누락·오기입 등 실수 발생 가능
- **불필요한 공수**: 전형마다 데이터 추출 → 정제 → 병합 → 검수 작업을 반복해야 하며, 담당자 의존도가 높음
- **현황 파악 지연**: 서면평가와 시험 결과를 따로 조회해야 하므로 실시간 전형 진행 현황 확인 불가
- **인수인계 리스크**: 운영 노하우가 특정 담당자에게 집중되어, 담당자 교체 시 운영 공백 발생 가능

---

### 1-2. aicapa.kr 실 운영 장애 (200명+ 동시 응시 기준)

#### ① 입력 불가 오류
- **현상**: 일부 수험자가 텍스트 입력이 되지 않거나 화면이 멈추는 현상 발생
- **원인 추정**: 답안 자동저장이 짧은 주기로 서버에 반복 요청 → DB 연결 포화
- **영향**: 수험자 공정성 침해, 재응시 처리 등 운영 부담 발생

#### ② 타이머 오작동
- **현상**: 시험 제한 시간이 남아있음에도 시험이 강제 종료됨
- **원인 추정**: 타이머를 클라이언트(브라우저) 기준으로 관리 → 탭 전환·백그라운드 전환 시 브라우저의 타이머 스로틀링 적용으로 오작동
- **영향**: 수험자 민원 발생, 운영 신뢰도 저하

#### ③ 웹캠 감시 시스템 불안정
- **현상**: 200명 웹캠 스트림 실시간 표출 중 끊김·감시 기능 오작동
- **원인 추정**: 전용 미디어 서버 없이 브라우저에서 직접 WebRTC 연결 처리 → 200개 이상 동시 스트림에서 클라이언트·서버 과부하
- **영향**: 부정행위 감시 신뢰도 저하, 운영 중 수동 대응 필요

### 1-3. evalu-lens 문제점

#### ① 평가 기준 일관성 부재
- 평가자마다 채점 기준이 달라 동일 지원서에 점수 편차 발생 가능
- 루브릭(항목별 채점 기준)이 명문화되지 않거나 시스템에 반영되지 않음

#### ② 피드백 미공개
- 불합격 사유가 지원자에게 전달되지 않음
- 결과: "왜 떨어졌나요?" 문의 전화·메일 대응

#### ③ 데이터 접근성
- 평가 결과 데이터를 시험 결과와 연계하여 보기 어려움
- 자체 DB 접근 불가 시 데이터 추출이 운영사에 의존


### 1-4. 접속·응시 행동 로그 부재

#### 현상
- 지원자가 플랫폼에 접속했는지, 시험 페이지를 열람했는지, 시험을 시작했는지 확인할 수단이 없었음
- 전형 종료 후 "이 사람이 시험을 봤나요?" 질문에 응시 기록(exam_attempts)으로만 확인 가능 — 시험을 열었다가 시작하지 않은 경우는 추적 불가

#### 원인
- aicapa.kr·evalu-lens 모두 행동 로그 테이블 없이 운영됨
- 결과 데이터(점수, 합격 여부)만 저장, 과정 데이터(언제 접속·열람·시작했는지) 미저장

#### 결과
- 미응시자 사후 파악 불가 → 재공지·구제 처리 기준 없음
- 이의제기 시 "시험을 보려 했으나 오류가 발생했다"는 주장에 대한 반증 자료 없음
- 운영 중 실시간 참여 현황 확인 불가

#### 해결책 (KBrain-AX)
-  테이블로 이벤트 기록: (시험 페이지 진입) / (시험 시작) / (제출)
- Supabase Pro DB에 영구 보관, 전형 종료 후에도 사후 조회 가능
- 관리자 화면에서 지원자별 행동 타임라인 확인

## 2. evalu-lens 평가 방식 개선

### 현재 방식의 문제

evalu-lens는 단일 Gemini LLM 호출로 지원서를 평가합니다. 이 방식의 근본적인 한계는:

- **블랙박스 판단**: "왜 이 점수인지" 설명 불가 → 불합격자 납득 어려움, 이의제기 대응 곤란
- **일관성 미보장**: 동일 지원서를 다시 돌리면 점수가 달라질 수 있음
- **학술적 근거 없음**: 평가 기준이 LLM의 사전학습 데이터에 의존, 검증된 이론 기반 아님
- **편향 통제 불가**: PII(이름·소속·성별 등) 노출 상태로 평가 시 편향 발생 가능

---

### 개선 방향: 루브릭 기반 설명 가능 평가

단순 LLM 판단에서 **논문·연구 기반 루브릭 + 2층 설명 구조**로 전환합니다.

#### 평가 구조

```
항목별 점수
  └─ 1층: 지원서 원문 인용 (어떤 문장이 근거인지)
  └─ 2층: 학술 근거 (어떤 이론·연구가 이 기준의 근거인지)
```

#### LLM-as-Judge 신뢰성·일관성 근거

에이전트의 핵심 설계 결정(낮은 temperature, N회 반복 검증, 캘리브레이션, 블라인드 모드)은 아래 연구들을 근거로 합니다.

| 연구 | 핵심 내용 | 에이전트 반영 |
|------|-----------|--------------|
| Zheng et al. (2023) "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena" *NeurIPS* | LLM 평가자가 명확한 기준과 구조화된 프롬프트를 받을 때 인간 평가자와 80%+ 일치율 달성. position bias·verbosity bias 존재 확인 | 루브릭 + 채점 앵커로 구조화, 블라인드 모드로 편향 최소화 |
| Liu et al. (2023) "G-Eval: NLG Evaluation using GPT-4 with Better Human Alignment" | Chain-of-thought 방식으로 평가 근거를 먼저 생성하게 하면 LLM 평가의 인간 일치율과 일관성이 유의미하게 향상 | 원문 인용(1층) → 학술 근거(2층) 순서로 설명 생성 |
| Kim et al. (2024) "Prometheus: Inducing Fine-grained Evaluation Capability in Language Models" | 세밀한 루브릭(scoring anchor)을 제공할수록 LLM 평가의 정밀도·재현성이 높아짐. 단순 점수 요청보다 기준 명시가 핵심 | 항목별 점수 구간(9-10점, 7-8점…) 앵커 방식 채택 |
| Wang et al. (2023) "Large Language Models are not Fair Evaluators" | LLM 평가자는 순서·길이·표현 방식에 편향됨. 단일 호출 결과를 그대로 신뢰하면 안 됨 | N회 반복 평가 + 표준편차 기반 일관성 검증, 캘리브레이션으로 보정 |

#### 평가 항목별 루브릭 이론 근거

| 평가 항목 | 적용 이론 |
|----------|----------|
| 지원동기 | Noe & Wilk (1993) — 교육 참여 동기가 학습 전이에 직접 영향. *Journal of Applied Psychology* |
| AI·데이터 활용 경력 | Kolb (1984) — 경험학습이론, 구체적 경험이 학습 효과 결정. *Experiential Learning* |
| 현안 연결성 | Broad & Newstrom (1992) — 교육 전이 환경 요인. *Transfer of Training* |
| 데이터 확보·활용 가능성 | Provost & Fawcett (2013) — 데이터 사이언스 역량의 핵심은 데이터 접근성. *Data Science for Business* |
| 기술 타당성 | Rogers (2003) — 기술 혁신 채택 요인. *Diffusion of Innovations* |
| 이행계획 구체성 | PMI PMBOK (2021) — 프로젝트 일정 관리 표준 |
| 기대효과 | Kirkpatrick (1994) — 교육 효과 4단계 평가 모델. *Evaluating Training Programs* |
| 블라인드 평가 | Schmidt & Hunter (1998) — 인사 선발 방법의 타당성과 편향 통제. *Psychological Bulletin* |

#### 추가 기능

- **블라인드 모드**: 이름·소속 등 PII 자동 마스킹 후 평가 → 편향 최소화
- **일관성 검증**: 동일 지원서 N회 반복 평가 → 표준편차로 안정성 확인
- **지원자 피드백 자동 생성**: 점수 제외, 강점·개선사항 중심 피드백 → "왜 떨어졌나요" 전화 방지
- **캘리브레이션**: 사람 평가 결과와 비교·보정하여 정확도 향상

> 위 방식은 자체 개발한 루브릭 기반 평가 에이전트(Claude API 기반)를 KBrain-AX에 통합하는 방식으로 구현 예정

## 3. 개선 방향 (KBrain-AX 전체)

| 기존 문제 | 개선 방향 |
|----------|----------|
| 서면평가·시험 시스템 분리 | 단일 플랫폼 통합, 전형 진행 현황 일원화 |
| DB 연결 포화로 인한 입력 오류 | 답안 로컬 임시저장 + 30초 주기 서버 동기화 |
| 클라이언트 타이머 오작동 | 서버 시작 시각 기준 경과시간 계산 방식으로 변경 |
| WebRTC 자체 구현 불안정 | Agora (전용 WebRTC 인프라) 또는 Zoom 병행 운영 |
| 주관적 평가 기준 | 루브릭 기반 AI 자동 평가, 항목별 점수화 |
| 불합격 사유 미공개 | 지원자 대시보드에 AI 평가 피드백 공개 (강점·개선사항) |
| 결과 수동 취합 | 서면평가·시험 결과 DB 자동 연계 |
| 접속·응시 로그 부재 | activity_logs 테이블로 이벤트 기록, 관리자 화면에서 지원자별 타임라인 확인 |

## 5. 기존 운영사에 요구할 항목

### 5-1. 역대 시험 문항

| 항목 | 요구 내용 | 형식 |
|------|-----------|------|
| 문제 본문 | 전체 문항 텍스트 | Excel / Word |
| 문제 유형 | 객관식·단답형·서술형 등 구분 | 동일 파일 내 컬럼 |
| 보기 | 객관식 선택지 전체 | 동일 파일 내 컬럼 |
| 정답 | 객관식·단답형 정답 | 동일 파일 내 컬럼 |
| 해설 | 정답 근거·풀이 (있는 경우) | 동일 파일 내 컬럼 |
| 난이도 | 상·중·하 또는 자체 기준 | 동일 파일 내 컬럼 |
| 출제 전형 | 어느 전형·회차에서 사용됐는지 | 동일 파일 내 컬럼 |

> 목적: KBrain-AX 문제 은행으로 바로 이관 가능한 형태로 확보

### 5-3. aicapa.kr 소스코드

evalu-lens 소스코드는 보유 중. **aicapa.kr 소스코드만 요청.**

| 항목 | 요구 내용 |
|------|-----------|
| 프론트엔드 | 시험 응시 화면, 타이머, 자동저장 로직 포함 전체 |
| 백엔드 | API 서버, DB 연결 구조, 자동저장 처리 로직 |
| 인프라 구성 | 서버 스펙, 배포 구성 |

**요구 근거**
- 실 운영 중 입력 오류·타이머 오작동·웹캠 불안정 장애가 발생했으며, **정확한 원인 파악을 위해 구현 코드 확인이 필요**
- 자동저장 주기, 타이머 구현 방식(클라이언트/서버 기준), WebRTC 처리 구조 등 장애 추정 원인을 직접 검증하기 위함
- KBrain-AX 설계 시 동일한 구조적 문제를 반복하지 않기 위한 참고 목적
